*Aligned with syllabus: “Association rules (continued): randomized algorithms and Map-Reduce implementation”; lab: simple programming exercises on sample datasets.*

---

# Task 1 – Conceptual questions

Answer in your own words:

1. Why does **frequent itemset mining** become expensive on very large transaction data?
2. What is the **MapReduce** programming model? Describe the roles of **map** and **reduce** with a simple example (not necessarily association rules).
3. What does it mean for an algorithm to be **randomized** in the context of mining? What trade-off is usually made (accuracy vs. …)?
4. Why can **counting candidate itemsets in parallel** help scale association rule discovery?

# Task 2 – From transactions to counts (pen-and-paper or code)

Use a **small** synthetic transaction dataset (at least **8** transactions, **≥4** items) of your own design.

1. List all transactions.
2. Using **minimum support = 50%** (of transactions), list **all frequent singletons** `{A}`, `{B}`, … with their support counts.
3. List **frequent pairs** (if any) at the same threshold.

# Task 3 – MapReduce-style word-count mindset

For the string:  
`big data big map reduce data map`

1. Write **mapper** output as `(key, 1)` pairs (split on spaces; lowercase).
2. Show **reducer** input grouped by key and the **final** `(word, count)` pairs.
3. In **2–3 sentences**, explain how this pattern relates to **counting item co-occurrences** in a larger pipeline (high level only).

# Task 4 – Mini implementation (pick one)

**Option A – Python:** Read a small CSV of transactions (one row = one basket, items in columns or comma-separated). Output **item counts** and **pair counts** for all pairs that appear in at least one basket (brute force is fine for small *n*).

**Option B – Pseudocode:** Write clear pseudocode for **parallel counting** of singletons where each mapper processes a subset of transactions and reducers sum counts.

Include **short** instructions to run Option A (dependencies, command).
